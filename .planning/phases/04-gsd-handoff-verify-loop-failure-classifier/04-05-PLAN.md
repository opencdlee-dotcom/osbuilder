**Section 3 — VERIFICATION.md Format:** Document the exact format that gsd_driver.py produces. Include the full template with section headers, placeholder text, and the "2-5 criteria" rule. The format must be presented as a fenced markdown code block.

**Section 4 — Falsifiability Rule:** State the rule: "Each criterion must be verifiable by a person with no code access — only browser, terminal, or observable application output." Provide this as a standalone callout.

**Section 5 — Forbidden Criterion Patterns:** Explicit list of what is NOT acceptable. Must include:
- `"All unit tests pass."` — explains: circular (tests can be wrong)
- `"pytest exits 0."` — explains: not an observable user behavior
- `"No errors appear in the logs."` — explains: not observable without code access, and normal logs contain non-error output too
- `"The API returns the correct response."` — explains: too vague; which API, which response, from where?
- `"The feature works correctly."` — explains: subjective, unverifiable

**Section 6 — Valid Criterion Examples (>= 5 examples):** Each example must follow the format:
```markdown
**[Observable behavior]:** [description]
- How to check: [concrete step with URL, command, or UI action]
- Acceptable because: [one sentence explaining why it's falsifiable]
```

Provide at least 5 examples covering different observable behaviors:

1. Route navigation: "User can navigate to `/login`, enter `test@example.com` / `password123`, and be redirected to `/dashboard`."
   - How to check: Open http://localhost:3000/login in a browser, enter the credentials, confirm redirect to /dashboard.

2. Data persistence: "Data entered in the form at `/todos/new` persists after a page refresh — the item appears in the list at `/todos`."
   - How to check: Create a to-do item, refresh the browser, confirm the item is visible in the list.

3. API liveness: "The health endpoint responds with HTTP 200."
   - How to check: Run `curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/api/health` — output must be `200`.

4. Error handling: "Submitting the login form with an invalid email shows an inline error message 'Invalid email address' without a page reload."
   - How to check: Enter `notanemail` in the email field, click Submit, confirm the error message appears inline.

5. Scaffolded project boots: "Running `pnpm dev` from the project directory starts the dev server and `http://localhost:3000` responds with HTTP 200."
   - How to check: Run `pnpm dev`, wait for "ready" output, then `curl -s -o /dev/null -w "%{http_code}" http://localhost:3000`.

6. File on disk: "After scaffolding, the file `.env.example` exists at the project root and contains at least one `=` character (it is not empty)."
   - How to check: Run `test -s .env.example && grep -q '=' .env.example && echo OK`.

**Section 7 — Count Rule:** "Each VERIFICATION.md must contain 2 to 5 criteria. Fewer than 2 means the phase was not verified. More than 5 means the phase scope was too large or criteria are not truly independent."

**Section 8 — Escalation Note:** "If gsd_driver.py cannot generate 2 valid falsifiable criteria for a phase (because the phase produced no observable user behavior), it must flag this as a scope review rather than generating un-falsifiable criteria."

The file should be approximately 60-90 lines. Do NOT include YAML frontmatter. Use `#` for the title, `>` for the scope disclaimer blockquote, `##` for sections. Keep it flat (no `###` unless needed for code blocks).
